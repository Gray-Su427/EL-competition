# 单用户单菜单条评价设计

## 背景

当前系统允许同一用户对同一道菜多次创建评价。这样会带来三个问题：

- 用户体验不一致：用户无法判断再次提交是“补充修改”还是“新增一条”
- 统计口径不稳定：同一用户的重复评价会抬高评论数并影响均分
- 数据治理薄弱：前端即使限制重复提交，后端仍可能写入重复记录

本次目标是把评价模型调整为“同一用户对同一道菜只能保留一条当前评价”，并允许该用户后续修改这条评价。

## 目标

- 每个 `user_id + dish_id` 最多存在一条当前评价
- 用户若已评价过某道菜，再次进入详情页时应直接走“修改已有评价”
- 修改评价时预填已有评分、文字、标签、图片
- 修改后评论列表仍只显示一条记录，并带“已编辑”标记
- 修改时允许保留旧图、删除旧图、补充新图

## 非目标

- 不保留评价历史版本
- 不新增管理员审核流
- 不支持对他人的评价进行编辑
- 不调整现有推荐、排序、AI 对话等无关模块

## 方案对比

### 方案 A：单条记录原地更新

首次评价时创建 `reviews` 记录，后续修改直接更新这条记录。

优点：

- 业务语义最清晰
- 查询最简单
- 对现有前后端侵入最小
- 能从根上避免重复评价

缺点：

- 不保留历史版本

### 方案 B：多版本记录，仅显示最新版本

每次修改都新增一条版本记录，查询时只显示最新版本。

优点：

- 可以追溯历史

缺点：

- 复杂度高
- 查询、统计、唯一性和图片清理都更麻烦

### 方案 C：仅前端限制重复创建

前端发现用户已评价后不再开放创建入口，但后端仍允许重复写入。

优点：

- 实现最快

缺点：

- 无法防止绕过前端后的脏数据
- 规则不可靠

## 决策

采用方案 A：单条记录原地更新。

## 数据模型设计

### reviews 表

新增或调整以下约束与字段：

- 为 `user_id + dish_id` 添加唯一约束
- 新增 `updated_at` 字段，记录最后一次编辑时间

行为约定：

- 首次创建评价时，`created_at` 由数据库生成
- 首次创建时 `updated_at` 为空
- 后续修改时写入 `updated_at`
- 前端以 `updated_at` 是否存在，或是否晚于 `created_at`，决定是否显示“已编辑”

数据兼容策略：

- 迁移前若数据库中已存在重复评价，需要在迁移脚本或启动修复逻辑中只保留一条
- 保留策略采用“同一用户同一菜品的最新一条”，其余重复记录删除
- 由于当前项目为 SQLite 开发数据，允许使用一次性数据整理逻辑完成收敛

## 后端接口设计

### 保留接口

`POST /api/reviews`

职责：

- 仅用于首次创建评价
- 如果同一用户已对该菜品创建过评价，返回冲突错误，并给出明确提示

返回建议：

- `201 Created`：首次创建成功
- `409 Conflict`：该用户已评价过该菜品，应改为编辑

### 新增接口

`PUT /api/reviews/{review_id}`

职责：

- 只允许评价作者更新自己的评价
- 支持更新评分、文字、标签
- 支持保留部分旧图、删除部分旧图、补传新图

请求建议：

- `rating`
- `content`
- `tags`
- `keep_images`：保留的旧图路径数组
- `images`：新上传图片数组

图片规则：

- 最终图片总数 = `keep_images.length + 新上传成功数量`
- 最终图片总数不能超过 3
- `keep_images` 中只能保留当前评价原本已有的图片
- 被移除的旧图文件应在更新成功后从磁盘删除

权限规则：

- 非作者返回 `403 Forbidden`
- 评价不存在返回 `404 Not Found`

### 可选增强接口

`GET /api/reviews/me?dish_id=xxx`

职责：

- 获取当前登录用户对指定菜品的评价，用于页面直接进入编辑态

备注：

- 若当前详情页本身已拉取了全部评论，也可以由前端从评论列表里识别自己的那条评价
- 若当前列表接口未来做匿名化或分页，这个接口会更稳

本次实现建议：

- 优先复用现有 `GET /api/reviews?dish_id=xxx`，前端从列表里匹配自己的评价
- 如发现前端无法稳定识别当前用户，再补 `GET /api/reviews/me`

## 前端交互设计

### 菜品详情页

新增判断：

- 当前用户已登录
- 当前用户是否已对该菜品评价

展示规则：

- 未评价：按钮文案为“写评价”
- 已评价：按钮文案为“修改评价”

点击行为：

- 未登录：跳转登录
- 已登录且未评价：打开空白评价表单
- 已登录且已评价：打开预填表单

### 评价表单

表单组件统一支持两种模式：

- 创建模式
- 编辑模式

编辑模式预填内容：

- `dishId`
- `rating`
- `content`
- `selectedTags`
- `existingImages`

图片区分为两类数据：

- 已有图片：来自当前评价，支持单张删除
- 新上传图片：本次补充上传，支持预览和删除

提交规则：

- 创建模式调用 `POST /api/reviews`
- 编辑模式调用 `PUT /api/reviews/{review_id}`

错误处理：

- 如果创建时后端返回 `409 Conflict`，前端提示“你已经评价过这道菜，已为你切换到修改模式”或直接刷新后进入编辑态
- 若图片超过 3 张或单张超限，沿用现有表单报错方式

### 评论卡片

每条评论仍只显示一张逻辑记录。

新增展示规则：

- 若 `updatedAt` 存在，则在时间旁增加“已编辑”文案

## 数据流

### 首次评价

1. 用户进入菜品详情页
2. 前端拉取评论列表
3. 未找到当前用户评价
4. 用户点击“写评价”
5. 提交 `POST /api/reviews`
6. 成功后刷新评论列表

### 再次评价

1. 用户进入菜品详情页
2. 前端拉取评论列表
3. 找到当前用户已有评价
4. 按钮显示“修改评价”
5. 用户点击后打开预填表单
6. 提交 `PUT /api/reviews/{review_id}`
7. 成功后刷新评论列表
8. 页面显示更新后的单条评价与“已编辑”标记

### 并发兜底

若前端状态过期，用户在创建模式下提交时后端发现已有评价：

1. 后端返回 `409 Conflict`
2. 前端提示该菜品已存在本人评价
3. 前端重新拉取评论列表
4. 重新进入编辑态

## 错误处理

- 菜品不存在：返回 `404`
- 评价不存在：返回 `404`
- 非作者修改：返回 `403`
- 重复创建：返回 `409`
- 图片格式不支持：返回 `400`
- 图片总数超过限制：返回 `400`
- 单图超过 5MB：返回 `400`
- 非法 `keep_images` 内容：返回 `400`

## 测试策略

### 后端

- 首次创建评价成功
- 同一用户对同一菜品重复创建返回 `409`
- 作者更新自己的评价成功
- 非作者更新返回 `403`
- 更新时保留部分旧图、删除部分旧图、补传新图成功
- 更新后最终图片总数超过 3 返回 `400`

### 前端

- 未评价时显示“写评价”
- 已评价时显示“修改评价”
- 编辑态表单正确预填
- 删除旧图与删除新图的交互正确
- 更新后评论卡片显示“已编辑”
- 创建时遇到 `409` 后能切到编辑态或至少给出明确提示

## 实施顺序

1. 后端模型增加唯一约束与编辑时间字段
2. 加入开发数据清理逻辑，消除已有重复评价
3. 后端新增更新接口并收紧创建接口
4. 前端识别“当前用户是否已评价”
5. 评价表单支持创建/编辑双模式
6. 评论卡片显示“已编辑”
7. 补充验证并手动回归

## 风险与缓解

风险：已有数据库中存在重复数据，直接加唯一约束会失败  
缓解：先执行去重，再创建唯一约束

风险：旧图删除与新图上传同时发生时，图片集合容易计算错误  
缓解：统一在后端按“保留旧图 + 新图”计算最终状态，再一次性写回

风险：前端只靠评论列表识别本人评价，可能依赖当前登录用户信息是否可用  
缓解：优先复用现有上下文；若识别不稳定，再补“获取我的该菜评价”接口
